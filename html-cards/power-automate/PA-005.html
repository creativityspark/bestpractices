PA-005 Name variables with typed camelCase

Variable names must use camelCase notation and a short type prefix so the data type is immediately obvious.

1
Guideline 1
Use a short type prefix such as str, int, bool, arr, obj, or flt at the start of each variable name.
2
Guideline 2
Use camelCase after the type prefix.
3
Guideline 3
Use descriptive names that clearly indicate the purpose of the variable.
4
Guideline 4
Avoid abbreviations, generic names, and meaningless one-character identifiers.
1
Rationale 1
Type prefixes make it immediately clear what kind of data the variable holds without inspecting its initialization.
2
Rationale 2
camelCase notation is consistent with the expression language used in Power Automate.
3
Rationale 3
Descriptive variable names make the flow self-documenting and easier to maintain.
4
Rationale 4
Clear variable names reduce confusion and errors when variables are referenced in expressions.
✓ Do
  • ✓ strCustomerFullName String variable with a descriptive name.
  • ✓ intNumberOfRetries Integer variable that clearly expresses its purpose.
  • ✓ boolIsEligibleForDiscount Boolean variable that reads as a condition.
  • ✓ arrPendingApprovals Array variable with a meaningful collection name.
  • ✓ objOrderDetails Object variable that clearly identifies its contents.
✗ Don't
  • ✗ x No indication of purpose or type.
  • ✗ temp Generic and unclear.
  • ✗ var1 Generic and unclear.
  • ✗ name Missing a type prefix and ambiguous.
  • ✗ n Single-character name with no meaning.
